SF Express Courier Manager System - Project Design Documentation#

1. System Overview#

SF Express Manager is an all-in-one logistics management platform tailored for corporate users to streamline domestic and international shipping workflows. It centralizes order creation, shipment approval, real-time tracking and automated financial reconciliation in one unified system. This tool effectively cuts operational costs, standardizes delivery management and greatly improves overall business logistics efficiency.

2. Project Parameters#

Role#

Full-stack Backend Developer, responsible for core business logic design, database architecture and API service development

Timeline &amp; Scope#

4-month enterprise-level internal production logistics management system, officially launched for corporate clients and branch office daily operation, not personal demo project

The Tech Stack#

  • Programming Language: Java (Spring Boot, Spring Cloud)

  • Database: MySQL, Redis

  • Message Queue: RabbitMQ

  • Middleware: Nginx, Elasticsearch

  • Infrastructure: Docker, Kubernetes, Linux Server

  • Dev Tools &amp; Others: Maven, Git, Swagger, MinIO

3. Problem Statement &amp; Business/Technical Constraints#

Problem Statement#

Enterprises lack a centralized platform to unify intra-departmental shipment management. Manual order entry, offline approval, scattered logistics tracking and manual financial reconciliation cause low operational efficiency, ambiguous cost allocation and poor shipping process standardization.

Business Constraints#

  1. Must support multi-level organizational role permissions and isolated data access across departments.

  2. Strictly comply with SF Express official shipping rules, billing standards and service interface specifications.

  3. Customizable approval workflows are mandatory to fit diverse enterprise management policies.

  4. Realize automatic expense apportionment, standardized billing and compliant financial statement export.

Technical Constraints#

  1. Ensure high concurrency stability to sustain bulk order submission during business peak hours.

  2. Achieve low-latency data synchronization via docking official SF Express open APIs.

  3. Optimize storage and query performance for large volumes of order and logistics trajectory data.

  4. Strictly protect sensitive user addresses, contact info and enterprise shipping confidential data.

  5. Be compatible with web, mobile mini-program and third-party enterprise office system integration.

4.2 Core Data Flow#

User submits shipping request → API Gateway verifies identity and routes the request → Shipment Order Service generates the order → Approval Flow Service triggers the audit process → Message Queue dispatches asynchronous tasks to sync with SF Express official API → Core business data is stored in MySQL, and logistics tracking data is indexed in Elasticsearch → Hot data (e.g., user info, common addresses) is cached in Redis → Financial Reconciliation Service generates statistics and supports compliant export.

4.3 Design Rationale#

1. Microservices Architecture Selection#

Decouple permission, order, approval, logistics and financial modules independently. This supports independent deployment, elastic scaling and separate maintenance, which perfectly matches enterprise tiered management demands and facilitates later function iteration and capacity expansion. If one module fails (e.g., logistics sync), it will not affect the normal operation of other core modules (e.g., order creation, approval).

2. SQL Database (MySQL) Over NoSQL#

The system involves strict transaction scenarios: order creation, approval status change, expense splitting and monthly account settlement. SQL databases provide strong ACID transaction consistency, standardized table structure and powerful multi-table joint query capability, which can accurately fulfill financial audit and data statistics requirements. NoSQL databases lack stable transaction support and structured query capabilities, making them unsuitable for financial-related core business where data accuracy and consistency are critical.

3. Message Queue (RabbitMQ) for Asynchronous Processing#

Real-time logistics status pulling, bulk waybill generation and batch expense statistics are all time-consuming non-blocking tasks. Using message queues decouples upstream business operations (e.g., order submission) and downstream third-party interface calls (e.g., SF Express API). This avoids interface timeout failures caused by synchronous blocking, improves system throughput, realizes peak traffic shaving, and guarantees task final consistency even under high concurrent submission pressure.

4. Cache Layer (Redis) Introduction#

Cache enterprise user information, common shipping addresses and real-time shipping price data to reduce repeated database queries and significantly shorten front-end response latency. For example, frequently accessed user permission data and common addresses are cached, avoiding frequent MySQL queries and improving the system’s response speed during peak hours.

5. Elasticsearch for Logistics Tracking#

Logistics tracking data is unstructured and requires fast full-text search (e.g., querying trajectory by order number or tracking number). Elasticsearch provides high-performance full-text search capabilities, which is more efficient than MySQL for large-scale unstructured data query, ensuring users can quickly obtain real-time logistics status.
